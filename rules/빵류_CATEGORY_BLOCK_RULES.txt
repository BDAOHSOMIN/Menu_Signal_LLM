## 공통 처리 원칙
- 프롬프트 예시에 있더라도 일반적으로 통용되는 메뉴 지식 기준으로 판단한다.
- 유럽식/미국식/멕시칸식 여부가 명확하지 않으면 cuisine block으로 강제 배정하지 않는다.
- 다양한 명칭이 많으므로 적극적으로 표기 변형 동의어도 고려하여 처리한다.
- BLOCK_PRIORITY를 위에서부터 적용한다.
- 특정 R block에 들어간 뒤에는 해당 block의 INTERNAL_PRIORITY를 위에서부터 적용한다.
- raw_generic_name은 2차 프롬프트에서 생성된 후보명이고, final_generic_name은 3차에서 확정하는 분석용 이름이다.
- final_generic_name 후보 목록은 “강제 통합 가능한 대표명 목록”이다.
- raw_generic_name이 block 안의 final_generic_name 후보로 명확히 통합 가능하면 반드시 해당 final_generic_name으로 통합한다.
- AI가 새로운 assigned_block이나 family_group을 임의 생성하면 안 된다.
- AI가 CATEGORY_BLOCK_RULES에 없는 넓은 final_generic_name을 임의 생성하면 안 된다.
- 빵류가 아닌 후보는 R98_OUT_OF_SCOPE로 보낸다.
- cuisine_signal은 block과 별도로 판단하되, 메뉴명 안에 명확한 문화권/업태 signal이 있을 때만 남긴다.
- cuisine_signal을 판단할 때 긴 유래 설명을 생성하지 않는다.
- 피자류, 버거류, 핫도그류는 큰 메뉴 형태 block이므로 먼저 처리한다.
- 피자류는 세트/플래터/조각/사각 형태가 붙어도 피자 본체가 명확하면 모두 피자류로 처리한다.
- 버거류는 세트/플래터/수제/더블/치즈/불고기 등 세부 표현이 붙어도 버거 본체가 명확하면 모두 버거류로 처리한다.
- 핫도그류는 세트/미니/감자/치즈/소시지 등 세부 표현이 붙어도 핫도그 본체가 명확하면 모두 핫도그류로 처리한다.
- 멕시칸식빵류, 유럽식빵류, 미국식빵류, 동남아빵류, 일식빵류, 중식빵류는 cuisine/업태 signal이 명확한 경우에만 처리한다.
- 유럽식빵류는 대중적으로 유럽/지중해식 bread로 인식 가능한 후보만 받는다.
- 미국식빵류는 기준이 애매할 수 있으므로, 미국식 signal이 명확한 후보를 모아 검수하기 위한 bucket으로 둔다.
- 샌드위치토스트류는 후순위 keyword fallback block이다.
- 샌드위치류는 clean_text 또는 raw_generic_name에 샌드위치/샌드/sandwich 키워드가 명확히 있을 때만 배정한다.
- 토스트류는 clean_text 또는 raw_generic_name에 토스트/toast 키워드가 명확히 있을 때만 배정한다.
- 샌드위치/토스트 키워드가 없으면 형태상 샌드위치처럼 보여도 R10으로 보내지 않는다.
- 디저트빵류는 count가 충분하고 대표성이 명확한 후보만 final_generic_name으로 통합한다.
- R12_디저트빵류 후보에도 들어가지 않는 잔여 디저트/브런치/일반베이커리 후보는 R12_디저트빵류에 배정하되 final_generic_name = raw_generic_name으로 유지한다.
- R97_빵류복합검토는 cuisine/block 충돌 또는 빵류 여부/대표명 판단이 어려운 후보만 보낸다.
- 판제로티/판체로티는 로티/로띠가 포함되어도 동남아 로티로 처리하지 않는다.
- 모찌리도후/모찌리브륄레는 모찌 간식으로 자동 통합하지 않는다.
- 칠리도그/치즈도그/감자도그는 핫도그 맥락이 명확하면 R03_핫도그류로 처리한다.
- 마르게리따는 마르게리타와 동일하게 피자류로 처리한다.

## cuisine_signal 간단 판단 가이드

- 멕시칸식: 타코, 퀘사디아, 브리또, 또띠아, 나초 등 멕시칸/텍스멕스 신호가 명확한 경우
- 유럽식: 나폴리/마르게리타/고르곤졸라 계열 피자, 잠봉뵈르, 파니니, 바게트, 치아바타, 부르스케타, 포카치아, 브리오슈 등 유럽/지중해 bread signal이 명확한 경우
- 미국식: 시카고/뉴욕/디트로이트/하와이안 피자, 루벤, 포보이, 내슈빌핫치킨, 아메리칸 등 미국 지역/업태 signal이 명확한 경우
- 동남아식: 반미, 로띠/로티, 등 동남아/남아시아 bread signal이 명확한 경우
- 일식: 산도, 타코야끼, 모찌, 센베 등 일본식 signal이 명확한 경우
- 중식: 꽃빵, 바오, 바오번 등 중화권 bread signal이 명확한 경우


## R98_OUT_OF_SCOPE

family_group = ""

목적:
빵류가 아니거나 다른 category가 더 적절한 후보를 분리한다.

매칭 조건:
- 국, 탕, 찌개, 전골, 볶음, 구이, 조림, 찜, 튀김, 전, 부침, 무침, 회처럼 빵류가 아닌 조리법 본체
- 밥, 덮밥, 볶음밥, 국밥, 면, 국수, 라멘, 우동, 파스타처럼 다른 주식 category 본체
- 샐러드, 스프, 죽, 음료, 주류처럼 빵류가 아닌 메뉴 본체
- 춘권처럼 튀김류에서 처리하는 것이 더 적절한 후보
- 치미창가가 튀김류 본체로 명확하게 등장하는 경우
- 스테이크 단독 또는 스테이크 본체가 강한 후보
- 소스명, 토핑명, 추가, 옵션, 비메뉴 설명문

출력 규칙:
- assigned_block = R98_OUT_OF_SCOPE
- final_generic_name = raw_generic_name
- family_group = ""
- normalization_actions = ["exclude_or_correction"]
- review_status = correction_candidate


## R01_피자류

family_group = 피자류

final_generic_name 후보:
- 피자

INTERNAL_PRIORITY:
1. 피자

bucket 판단:
- 피자류는 피자 업태 signal이 명확한 후보를 처리한다.
- 피자 후보는 모두 final_generic_name = 피자로 통합한다.
- 피자세트, 피자플래터, 조각피자, 사각피자 피자 본체가 명확하면 피자로 통합한다.
- 피자 단어가 없어도 마르게리타, 고르곤졸라, 페퍼로니, 콰트로포르마지오, 디아볼라, 하와이안, 시카고, 뉴욕, 나폴리처럼 피자 메뉴로 명확하면 피자류로 보낼 수 있다.
- 세부 피자 스타일과 토핑은 attribute에 남긴다.
- cuisine_signal은 나폴리/마르게리타/고르곤졸라 계열이면 유럽식, 시카고/뉴욕/디트로이트/하와이안 계열이면 미국식처럼 명확한 경우에만 남긴다.

통합 규칙:
- 피자, 치즈피자, 페퍼로니피자, 콤비네이션피자, 불고기피자, 포테이토피자, 쉬림프피자, 고르곤졸라피자, 마르게리타피자, 하와이안피자, 루꼴라피자, 시카고피자, 뉴욕피자, 나폴리피자, 화덕피자, 사각피자, 조각피자, 피자세트, 피자플래터, 피자빵
  → final_generic_name = 피자

주의 규칙:
- 마르게리따, 마르게리따피자, 핏자, 피짜, 콰트로포르마지, 콰트로 포마지오, 콰트로 포마쥬, 콰뜨로 포르마주, 깔조네, 칼조네, 칼쪼네, 스트롬볼리, 핀사
  → final_generic_name = 피자
- 피자 단어가 없어도 마르게리타, 마르게리따, 고르곤졸라, 페퍼로니도 피자로 판단한다. (추론 필요)
- 페퍼로니 단독, 깔조네, 스트롬볼리, 핀사는 유럽식빵 미국식빵에 들어가기 전에 피자쪽에서 적극적으로 검토를 먼저 하고 넣는다.
- 피자의 종류는 유럽식빵,미국식빵이 아니라 피자 안에서 모두 처리 되어야 한다.
- 스테이크피자는 피자 본체가 명확하므로 R01_피자류로 처리하고, 스테이크는 attribute에 남긴다.

attribute 규칙:
- 치즈, 페퍼로니, 콤비네이션, 불고기, 포테이토, 쉬림프, 고르곤졸라, 마르게리타, 하와이안, 루꼴라, 시카고, 뉴욕, 나폴리, 화덕, 세트, 플래터 등 세부 스타일은 attribute에 남긴다.


## R02_버거류

family_group = 버거류

final_generic_name 후보:
- 버거

INTERNAL_PRIORITY:
1. 버거

bucket 판단:
- 버거류는 햄버거/버거 업태 signal이 명확한 후보를 처리한다.
- 치즈버거, 불고기버거, 수제버거, 치킨버거, 새우버거, 베이컨버거 등 세부 종류가 달라도 final_generic_name = 버거로 통합한다.
- 버거세트, 버거플래터도 버거 본체가 명확하면 final_generic_name = 버거로 통합한다.
- 세부 패티/스타일은 attribute에 남긴다.

통합 규칙:
- 버거, 햄버거, 치즈버거, 불고기버거, 수제버거, 치킨버거, 새우버거, 베이컨버거, 더블버거, 클래식버거, 하우스버거, 패티버거, 버거세트, 버거플래터
  → final_generic_name = 버거

주의 규칙:
- 라이스버거처럼 밥류 signal이 강하면 R97 또는 R98로 보낼 수 있다.
- 치킨스테이크버거, 뉴욕스테이크버거처럼 버거 본체가 명확하면 R02_버거류로 처리하고 스테이크는 attribute에 남긴다.

attribute 규칙:
- 치즈, 불고기, 수제, 치킨, 새우, 베이컨, 더블, 클래식, 패티, 세트, 플래터 등 세부 정보는 attribute에 남긴다.

family_group = 핫도그류

통합 규칙:
- 핫도그, 미니핫도그, 감자핫도그, 치즈핫도그, 소시지핫도그, 수제핫도그, 크리스피핫도그, hotdog, hot dog, 콘도그, 코니도그, 칠리도그, 치즈도그, 감자도그, 핫도그세트
  → final_generic_name = 핫도그

주의 규칙:
- `도그` 단독은 핫도그로 자동 통합하지 않고 R97 또는 R99로 보낸다.

attribute 규칙:
- 미니, 감자, 치즈, 소시지, 수제, 크리스피, 콘도그, 세트 등 세부 정보는 attribute에 남긴다.

빵류 안에는 남을 수 있지만, 기존 R01~R12의 family_group 또는 final_generic_name 후보로 자동 통합하면 의미 손실이나 category 혼동이 생길 수 있는 후보를 검토용으로 분리한다.

매칭 조건:
- 여러 업태/cuisine signal이 결합된 후보
- 디저트/브런치/일반베이커리 후보 중 R12_디저트빵류에도 안정적으로 넣기 어려운 후보
- 샌드위치/토스트와 특수 cuisine 경계 후보
- 빵류와 다른 category 경계가 애매한 후보
- final_generic_name 후보로 자동 통합하면 의미 손실이 큰 후보
- 브랜드명 또는 고유명처럼 보이지만 빵류 본체는 남아 있는 후보

제외 조건:
- 피자류 signal이 명확하면 R01
- 버거류 signal이 명확하면 R02
- 핫도그류 signal이 명확하면 R03
- 멕시칸식빵류 signal이 명확하면 R04
- 유럽식빵류 signal이 명확하면 R05
- 미국식빵류 signal이 명확하면 R06
- 동남아빵류 signal이 명확하면 R07
- 일식빵류 signal이 명확하면 R08
- 중식빵류 signal이 명확하면 R09
- 일반 샌드위치토스트 keyword가 명확하면 R10
- 디저트빵류 fallback으로 받을 수 있으면 R12
- 빵류가 아님이 명확하면 R98
- 빵류 여부 자체가 불명확하면 R99

출력 규칙:
- assigned_block = R97_빵류복합검토
- final_generic_name = raw_generic_name
- family_group = 빵류복합검토
- normalization_actions = ["broad_merge_review"]
- review_status = needs_review


## R99_REVIEW

family_group = ""

목적:
빵류 여부, 음식명 여부, 또는 적용 가능한 block 자체가 불명확한 후보를 최종 검토한다.

매칭 조건:
- 어떤 block에도 안정적으로 들어가지 않는 경우
- raw_generic_name만으로 빵류인지 알 수 없는 경우
- 음식명인지 옵션/소스/재료명인지 불명확한 경우
- 오타가 심해 빵류 여부 판단이 어려운 후보
- 브랜드명 또는 업장 고유명처럼 보이나 빵류 본체가 드러나지 않는 후보
- 재료가 불명확하거나 어느 빵류 축에도 확실히 속하지 않는 후보

출력 규칙:
- assigned_block = R99_REVIEW
- final_generic_name = raw_generic_name
- family_group = ""
- normalization_actions = ["broad_merge_review"]
- review_status = needs_review
